### 一、背景

我们以一个正常的网站开发为例。

通常需要前端开发人员，后端开发人员，运维，等等职位。

我们今天研究一下，如何提高前端的工作效率。

众所周知，一个人如果只干一件事，和一个人干很多事情。必然是只干一件事的效率更高。

所以，有了一个思考。

### 二、分析

一个项目，不可能只有一个功能，一个稍大的项目，几百上千的功能点，太正常了。

在前端的开发过程中，前端需要处理下面几件事：

- 工程化

  这个一般是架构师，或者实际干着架构师职能的其他职位人员(比如开发组长)来实现。

  配置好项目的基础设施: 技术选型，语法检查，提交推送的钩子处理，项目打包配置，多语言配置，部署流程等等。

  也就是说，一个项目把业务开发去掉，作为黑盒子，剩下的部分，基本都都要概括进去处理。

- 数据处理

  这个其实不难理解，就是要沟通后端接口，拿到所有的接口配置。再根据业务需求，把后端的数据改造成前端需要的样子。

  在数据返还给后端的时候，把数据改造成后端需要的样子。

  在后端开发完成接口之前，向 UI 层，提供用于渲染的 mock 数据。

  但是实际上，数据层的工作，还包括了: 数据的缓存，数据的清理，数据的校验，数据的各种转换处理流程。

  而且还有一些需要和后端`沟通`，数据适不适合放在前端。

  因为有很多后端，根本不考虑实际业务，只考虑原子化，可复用。。。彻底走上了歪路。

  当然了，也有很多新手前端，如果后端提供的数据结构不是和业务需要的一模一样，他就不乐意，一定要和后端吵一架。。。这种是真的很不应该。

  > 我们的任何设计，都要为了提升整体效率，而不是为了某个点的效率达到极致。如果因为一个点的提升，拉低了整体的效率，那么这种设计就是负面的。
  >
  > 其实，如果用马斯克提出的第一性原理来思考，那么很多事情就很容易理解了。

- ui还原

  主要是，拿到数据，把数据套进 UI。

  使用各种样式库，比如 tailwind，scss等来实现设计搞的样式需求。

  还有各种 UI 组件库引入，实现交互稿里面的业务逻辑。

  有些还需要根据设计，去匹配各种平台，各种客户端。

  完事了调用数据层提供的方法，把数据回传给后端。

我们正常在开发的时候，通常的流程（大概流程）：

- 分析业务逻辑，得到前端需要的数据结构

- 和后端对齐设计，得到最终的数据接口设计。

- 根据数据接口设计，mock 渲染的数据

- 根据设计搞和交互稿，完成 UI 层的代码。

- 根据写完的代码，编写测试用例。

- 对接后台的数据接口，让业务流程跑通。也就是所谓的前后端联调。

主要流程就是上面的了。

### 三、结果

结合我开始提到的，单一事情可以明显提升工作效率。那么，就可以得出，前端这边分成 2 个岗位：一个是 UI 岗，一个是数据岗。

主要的开发流，程演变成:

- 步骤一:

  - UI岗和数据岗，一起分析业务逻辑，得到前端需要的数据结构。
  - 后端分析业务逻辑，得到后端能提供的数据接口。

- 步骤二:

  - UI岗，根据UI设计，生成 mock 数据，开始实现静态的UI界面。
  - 数据岗和后端开始沟通确定最终的数据结构。

- 步骤三:

  - 数据岗，根据最终的数据结构，设计前后端的数据转换逻辑。
  - 后端正式提供，可用的数据给前端。（按功能分批提供）
  - 数据岗，测试后端接口的准确性。

- 步骤四：

  - 数据岗，提供给 UI 岗，不同功能的数据处理方法。
  - UI 岗，调用数据岗提供的方法，实现最终的交互逻辑。

- 步骤五：

  - 前后端分别针对调试正常的接口、UI，编写测试用例。（只有稳定的代码，才需要编写测试）

这是一个理想化的目标，让一个人只做一类事，可以把这类事情做的更深入更高效。

目的就是最大化的提升一个人的工作效率，这种方式，在面对小项目的时候，没有什么优势，甚至会降低效率。

但是，遇到大型项目的时候，成千上万的功能要开发，这种岗位设计，就会最大限度的提升人效。

当然了，这只是一种假设，没有经过实践，只是从第一性原理来分析，一个可能的高人效岗位方案。
